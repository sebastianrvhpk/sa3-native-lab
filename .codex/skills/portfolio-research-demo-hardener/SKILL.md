# Portfolio Research Demo Hardener

## Overview

Use this skill when a research prototype needs to communicate well to other engineers, researchers, designers, or hiring teams. The goal is not to productize prematurely. The goal is to make the work legible, runnable, honest, and technically impressive.

## Demo Hardening Workflow

1. State the research object.
   - What system is being explored?
   - What model, latent space, operator, or workflow does the project reveal?
   - What is confirmed, what is experimental, and what is unknown?

2. Create a runnable path.
   - One command or notebook path for setup.
   - One demo dataset or small fixture.
   - One deterministic example where possible.
   - Clear hardware assumptions.

3. Show the architecture.
   - Frontend, backend, model runtime, artifact storage, and evaluation loop.
   - Include diagrams only if they explain real data flow.
   - Keep stack choices tied to concrete responsibilities.

4. Shape the interaction story.
   - What can the user do in 60 seconds?
   - What can a deeper reviewer inspect in 10 minutes?
   - What artifact proves the system did something real?
   - What tradeoffs are intentionally left open?

5. Harden the engineering signal.
   - TypeScript domain models and schemas.
   - Reusable components only where they reduce real duplication.
   - Tests around data transforms, API contracts, and critical UI state.
   - Screenshots or short capture of the working flow.
   - README or case-study notes that separate research from product.

6. Avoid false polish.
   - Do not hide limitations.
   - Do not claim model capabilities that are not measured.
   - Do not turn the interface into a fake SaaS dashboard.
   - Preserve the open-ended research mindset.

## Role Signals

Make sure the project can demonstrate:

- React and TypeScript fluency.
- Component architecture and state management judgment.
- Typed API and schema thinking.
- Creative AI domain understanding.
- Ambiguous research-to-interface translation.
- Testing, debugging, and performance awareness.
- Visual and interaction design taste.
- Clear ownership and iteration history.

Use `references/role-signal-map.md` when deciding what to emphasize for a specific role.
